== Overview

NOTE: **Workshop pacing (~4 hours hands-on)** — Overview-only (~8 min) covers the problem statement and Vault UI. Hands-on adds path inspection and facilitator secret injection.

Factory and AI workloads need credentials: MaaS LLM tokens, registry pull secrets, Kuadrant API keys, TLS material. Storing them in Git — even private repos — creates **rotation pain**, **fork leaks**, and **audit gaps**. Production teams centralize secrets in a vault and let operators sync into clusters at runtime.

This platform deploys **HashiCorp Vault** on the hub (`hashicorp-vault` Validated Pattern chart) and **External Secrets Operator for Red Hat OpenShift** on hub and spokes. Together they implement: *declare the sync in Git, keep the value in Vault*.

=== Why Vault on the hub?

[cols="2,3", options="header"]
|===
| Capability | Workshop example

| **Central store**
| One place for `secret/workshop/maas` instead of duplicating Bearer tokens in five namespaces

| **Policy tiers**
| `admin` read/write workshop paths; `user1` read-only — mirrors OpenShift RBAC

| **Audit**
| Who read or wrote a path (complements ACS runtime policies from module 20)

| **Rotation**
| Update KV once; ESO refresh pushes new data to Kubernetes Secrets
|===

Vault runs in namespace `vault`. OpenShift Console → **Platform Hub-Spoke** → **Vault**.

IMPORTANT: Use `https://vault-vault.%HUB_DOMAIN%/ui/` — the route root returns HTTP **307**.

=== Documented KV paths

ConfigMap `workshop-api-vault-paths` in `workshop-kuadrant-apis` lists paths the platform expects:

[source,yaml]
----
# charts/all/workshop-kuadrant-apis/templates/vault-paths.yaml (excerpt)
# secret/workshop/maas              — upstream MaaS Bearer (AI Gateway)
# secret/workshop/kuadrant/<user>/<api-product> — optional Kuadrant key mirror
----

Inspect in the cluster:

[source,bash]
----
oc get cm workshop-api-vault-paths -n workshop-kuadrant-apis -o yaml
----

Facilitator example — store MaaS token in Vault (never paste keys in Showroom or Git):

[source,bash]
----
oc exec -n vault vault-0 -- vault kv put secret/workshop/maas api-key="<maas-token>"
----

== Why ExternalSecret matters

A Kubernetes **`Secret`** holds base64 data consumed by pods. An **`ExternalSecret`** (ESO CR) tells the operator: *create or update that Secret by reading an external store* — here, Vault KV.

.Without ExternalSecret — anti-patterns
[cols="1,2", options="header"]
|===
| Approach | Problem

| `apiKey: sk-...` in `values.yaml`
| Committed to Git; visible in forks and CI logs

| `oc create secret` in runbooks only
| Not declarative; drift between hub namespaces; hard to reproduce RHDP clusters

| Same MaaS Bearer in every Deployment
| One leak compromises NeuroFace, Lightspeed, OpenShift AI, and the gateway
|===

.With ExternalSecret — target pattern
[cols="1,2", options="header"]
|===
| Benefit | How

| **GitOps-safe**
| Git contains the `ExternalSecret` manifest; Vault holds values

| **Least privilege**
| Apps mount only their K8s Secret; users keep Kuadrant **APIKEY** (module 20)

| **Refresh**
| `refreshInterval` re-syncs when facilitators rotate Vault KV

| **Fleet**
| Hub Vault + `ClusterSecretStore` can serve spoke namespaces (same pattern east/west)
|===

Example — AI Gateway upstream credential (target state):

[source,yaml]
----
apiVersion: external-secrets.io/v1beta1
kind: ExternalSecret
metadata:
  name: ai-maas-upstream
  namespace: ai-gateway-system
spec:
  refreshInterval: 1h
  secretStoreRef:
    name: vault-workshop
    kind: ClusterSecretStore
  target:
    name: ai-maas-upstream-credentials
  data:
    - secretKey: api-key
      remoteRef:
        key: secret/data/workshop/maas
        property: api-key
----

Kuadrant AuthPolicy on the AI Gateway reads `ai-maas-upstream-credentials` to call RHDP MaaS. Workshop users never see this Bearer — they use **APIKEY** from Developer Hub (xref:23-ai-gateway.adoc[module 23]).

=== Secret flow (hub AI stack)

[source,text]
----
Facilitator → Vault KV (secret/workshop/maas)
                    ↓ ESO ExternalSecret
              K8s Secret (ai-maas-upstream-credentials)
                    ↓
              AI Gateway injects Bearer → MaaS
                    ↑
         User curl with Authorization: APIKEY
----

Same model applies to:

* **NeuroFace** — `neuroface-maas-api-key` (xref:27-neuroface.adoc[module 27])
* **OpenShift AI** — `openshift-ai-maas-credentials` in `maas-workshop` (xref:22-openshift-ai.adoc[module 22])
* **Lightspeed** — `llama-stack-secrets` in `developer-hub` (xref:24-mcp-gateway.adoc[module 24])

=== Workshop day-2 (today)

RHDP labs often inject keys post-install when Vault/ESO is not fully wired:

[source,bash]
----
export MAAS_KEY_LLAMA='sk-...'
bash scripts/apply-maas-secrets.sh
----

The script creates or patches Secrets in multiple namespaces **without printing keys**. It is acceptable for facilitators; production should replace it with Vault paths + `ExternalSecret` resources tracked in Git.

WARNING: Never commit `sk-*` keys, `values-secret.yaml`, or Vault root tokens. Use `values-secret.yaml.template` as the only tracked reference.

=== 3. Connect to module 20 and 23

. In Developer Hub you already requested a Kuadrant **APIKEY** (module 20).
. That key authorizes *you* at the gateway edge.

  -H 'Content-Type: application/json' \
